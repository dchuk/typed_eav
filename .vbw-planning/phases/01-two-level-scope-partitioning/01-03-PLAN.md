<objective>
Extend `Field::Base` to participate in the three-key partition `(entity_type, scope, parent_scope)`. Update the `for_entity` scope to filter on both scope keys, update the AR uniqueness validator on `name` to include `parent_scope` in its scope, and add a new model-level invariant validator that rejects rows with `parent_scope` present when `scope` is nil (orphan-parent rejection).

Per CONTEXT.md, we inline-duplicate this logic across `Field::Base` and `Section` (plan 04) — no `Scopable` concern this phase. The two implementations are intentionally symmetric so future extraction is mechanical when the rule of three triggers.

The new model-level invariant complements the cross-(scope, parent_scope) write guard added to `Value#validate_field_scope_matches_entity` in plan 05. Two layers of defense:
  - Field-level (this plan + plan 04 for Section): a Field row CANNOT be saved with parent_scope set when scope is nil. Caught at field-definition write time.
  - Value-level (plan 05): a Value cannot attach a Field whose `(scope, parent_scope)` doesn't match the entity's `(typed_eav_scope, typed_eav_parent_scope)`. Caught at value write time.

The Field-level invariant is the canonical guard — once enforced, no orphan-parent field rows can exist, which means the Value-level check never has to handle an orphan-field case.
</objective>

Replace the existing `for_entity` scope with a two-kwarg version:

  scope :for_entity, lambda { |entity_type, scope: nil, parent_scope: nil|
    where(
      entity_type: entity_type,
      scope: [scope, nil].uniq,
      parent_scope: [parent_scope, nil].uniq,
    )
  }

Semantics:
- `for_entity(et)` → returns global-and-global rows (scope nil, parent_scope nil). Identical to today's behavior because `parent_scope` defaults to nil and the resulting `[nil, nil].uniq == [nil]` filter selects rows where parent_scope IS NULL.
- `for_entity(et, scope: "t1")` → returns rows in scope "t1" plus globals. parent_scope still defaults to nil → returns rows with parent_scope nil (the orphan-parent invariant guarantees no scoped row ever has parent_scope present without a scope, so this is the natural superset for the "single-scope user" path).
- `for_entity(et, scope: "t1", parent_scope: "p1")` → returns rows whose (scope, parent_scope) is one of: ("t1", "p1"), ("t1", nil), (nil, nil). The (nil, "p1") tuple cannot exist by orphan-parent invariant.
- `for_entity(et, scope: nil, parent_scope: "p1")` → returns rows where scope is nil and parent_scope IS NULL or "p1". By orphan-parent invariant, scope=nil implies parent_scope=nil, so this collapses to global-and-global rows only. (parent_scope: "p1" is dead-letter when scope is nil — passing it is harmless but logically equivalent to omitting it.)

Inline-comment the for_entity body explaining the three-key set semantics so future readers understand why the orphan-parent invariant lets us write `[parent_scope, nil].uniq` unconditionally.

Important: keep the public method signature backwards-compatible. Single-scope users calling `for_entity(et, scope: "t1")` continue to work without code changes — `parent_scope` defaults to nil and the where-clause behaves correctly.

Add a new validation declaration in the validations block:

  validate :validate_parent_scope_invariant

And implement the private method (place it next to `validate_default_value` and `validate_type_allowed_for_entity` in the private section):

  # Orphan-parent invariant: when `scope` is nil (global field), `parent_scope`
  # MUST also be nil. A "global parent_scope" makes no semantic sense — a
  # field that's global across tenants cannot be partitioned within one tenant.
  # The paired-partial unique indexes added in the parent_scope migration
  # rely on this invariant: the global partial omits parent_scope from its
  # column list because parent_scope is always NULL when scope is NULL.
  def validate_parent_scope_invariant
    return if parent_scope.blank?
    return if scope.present?

    errors.add(:parent_scope, "cannot be set when scope is blank")
  end

Why blank? rather than nil?: empty-string parent_scope is not meaningful either (it would slip past `parent_scope.nil?` in older Ruby idioms); `blank?` rejects both. Same for `scope.present?` (not `!scope.nil?`).

This validator complements the Value-side `validate_field_scope_matches_entity` (extended in plan 05) which checks the (scope, parent_scope) match between Field and entity at Value write time. The Field-level guard prevents orphan rows from ever existing.
