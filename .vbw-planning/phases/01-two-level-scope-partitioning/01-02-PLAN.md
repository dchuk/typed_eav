<objective>
Convert the ambient scope resolver chain from scalar-returning to tuple-returning. After this plan: `TypedEAV.current_scope` returns `[scope_string_or_nil, parent_scope_string_or_nil]`, `TypedEAV.with_scope` accepts a tuple form, and `Config::DEFAULT_SCOPE_RESOLVER` returns a two-element array. This is the breaking change at the heart of Phase 1 — every caller of `current_scope` will need updating in plan 05.

Per CONTEXT.md, the tuple-returning resolver is a hard breaking change. Custom `Config.scope_resolver` callables in consumer apps must update on upgrade, even if they aren't using `parent_scope`. We do NOT auto-coerce non-array returns (the rejected BC-shim path); a custom resolver returning a bare scalar raises `ArgumentError` at the resolver-fallback branch inside `TypedEAV.current_scope`, BEFORE the value reaches `normalize_scope` (which would otherwise BC-coerce it and silently swallow the contract violation). The CHANGELOG entry and README migration note land in plan 07.

Critical design split — the with_scope BC path vs. the resolver-callable strict-contract path are TWO different surfaces:

1. **`with_scope(value)` block (public API, BC-preserving):** scalar input is accepted and coerced to `[normalized_scalar, nil]`. This is `normalize_scope`'s permissive shape, applied ON PUSH inside `with_scope`. The public block API has accepted scalars since v0.1.x and continues to.

2. **`Config.scope_resolver` callable return (strict contract):** the resolver's RAW return value is inspected BEFORE any normalization. Acceptable shapes: `nil` (resolver opted out) or a 2-element Array. Anything else — including a bare scalar — raises `ArgumentError` immediately. This is the breaking change.

The implementation enforces the split by branching INSIDE `current_scope`'s resolver-fallback path: it does NOT call `normalize_scope` on the resolver result. Instead it validates shape first, then calls `normalize_scope` only on each tuple element (or returns nil). This keeps `normalize_scope` itself permissive (so `with_scope("t1")` still BC-coerces) but removes it as the silent-miscoercion chokepoint that would otherwise hide a bad resolver.

Plan 05's `resolve_scope` will rely on `current_scope` returning `nil | [a, b]` — never a bare scalar. The strict-contract raise lives in `current_scope`, not `resolve_scope`.
</objective>

Two distinct concerns implemented in this task. Keep them visually separated in the source so future readers see why one path coerces and the other rejects.

(A) `TypedEAV.with_scope(value)` — public block API, BC-preserving. Single-arg signature is unchanged. Internally, push the value through `normalize_scope` BEFORE pushing on the stack so the stack always stores `[scope, parent_scope]` tuples (or `nil`).

  def with_scope(value)
    stack = (Thread.current[THREAD_SCOPE_STACK] ||= [])
    stack.push(normalize_scope(value))
    yield
  ensure
    stack&.pop
  end

`with_scope("t1")` pushes `["t1", nil]` (BC), `with_scope(["t1", "ps1"])` pushes `["t1", "ps1"]`, `with_scope(ar_record)` pushes `[ar_record.id.to_s, nil]`. The `ensure` pop is unchanged.

(B) `TypedEAV.current_scope` — strict on the resolver-callable return value. The resolver's RAW return MUST be `nil` or a 2-element Array; any other shape raises `ArgumentError` BEFORE `normalize_scope` is applied. This is the chokepoint that makes the breaking change visible.

  def current_scope
    return nil if Thread.current[THREAD_UNSCOPED]

    stack = Thread.current[THREAD_SCOPE_STACK]
    # Stack already holds normalized tuples (with_scope normalized on push).
    return stack.last if stack.present?

    # Resolver-callable strict-contract path. We deliberately do NOT pass the
    # raw return through normalize_scope, because normalize_scope is permissive
    # (scalar → [scalar, nil]) for with_scope's BC and would silently swallow a
    # contract violation by a custom resolver.
    raw = Config.scope_resolver&.call
    return nil if raw.nil?

    unless raw.is_a?(Array) && raw.size == 2
      raise ArgumentError,
            "TypedEAV.config.scope_resolver must return a 2-element [scope, parent_scope] " \
            "Array (or nil). Got: #{raw.inspect}. " \
            "v0.1.x resolvers returning a bare scalar must update — see CHANGELOG."
    end

    # Tuple shape verified — normalize each slot through the same scalar coercion
    # that normalize_scope uses. (We call normalize_scope here with the verified
    # 2-element Array, which is one of normalize_scope's accepted input shapes
    # and produces the canonical [String|nil, String|nil] tuple.)
    normalize_scope(raw)
  end

Why the strict check lives here rather than in plan 05's `resolve_scope`: `resolve_scope` consumes `current_scope`'s return value, so a permissive `current_scope` (one that ran `normalize_scope` over a bare-scalar resolver return) would coerce `"scalar"` → `["scalar", nil]` BEFORE `resolve_scope` ever saw it. The contract violation would be unreachable. By raising HERE, we guarantee `current_scope`'s return is always `nil` or a verified tuple, and downstream code (plan 05) can rely on that without re-checking shape.

Document on `with_scope` (in YARD): you may pass a scalar (back-compatible "scope only"), an AR record (id is normalized to string), or a `[scope, parent_scope]` Array. Keep the BC pattern obvious in the docstring.

Document on `current_scope` (in YARD): returns `nil` (no ambient scope) or a 2-element `[String|nil, String|nil]` tuple. Custom `Config.scope_resolver` callables MUST return one of those shapes; a bare scalar return raises `ArgumentError`. This contract is hard-enforced and is the user-visible breaking change in this phase.
