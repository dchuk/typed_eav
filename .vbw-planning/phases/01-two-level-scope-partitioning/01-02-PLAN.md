<objective>
Convert the ambient scope resolver chain from scalar-returning to tuple-returning. After this plan: `TypedEAV.current_scope` returns `[scope_string_or_nil, parent_scope_string_or_nil]`, `TypedEAV.with_scope` accepts a tuple form, and `Config::DEFAULT_SCOPE_RESOLVER` returns a two-element array. This is the breaking change at the heart of Phase 1 — every caller of `current_scope` will need updating in plan 05.

Per CONTEXT.md, the tuple-returning resolver is a hard breaking change. Custom `Config.scope_resolver` callables in consumer apps must update on upgrade, even if they aren't using `parent_scope`. We do NOT auto-coerce non-array returns (the rejected BC-shim path); a custom resolver returning a bare scalar will surface a clear error in plan 05's `resolve_scope`. The CHANGELOG entry and README migration note land in plan 07.

Backwards compatibility for the `with_scope(value)` block API is preserved: a single-arg call still means "set scope=value, parent_scope=nil" — the public block API itself is NOT a breaking change. Only the resolver-callable contract is.
</objective>

  <name>Update normalize_scope to accept and return tuples</name>
  <files>
    lib/typed_eav.rb
  </files>
  <action>
Rewrite `TypedEAV.normalize_scope` so it always returns a two-element tuple `[scope, parent_scope]` where each element is a String or nil. Accepted inputs:

  - `nil` → return `nil` (preserves "nothing resolved" sentinel for resolver chain)
  - A two-element Array `[a, b]` → return `[normalize_one(a), normalize_one(b)]`
  - Any other value (scalar / AR record) → return `[normalize_one(value), nil]` (preserves single-arg with_scope BC)

Extract a private helper method `normalize_one(value)` that does today's scalar coercion: `value.respond_to?(:id) ? value.id.to_s : value.to_s` (and returns nil for nil input).

Why an Array form is accepted as input: callers that already have a tuple (a custom resolver, a future `with_scope([s, ps])` call) should be able to pass it through unchanged.

Why the bare scalar passthrough returns `[normalized, nil]`: this is the BC path for `with_scope(scalar)` — single-arg block usage continues to mean "scope=scalar, parent_scope=nil". This is NOT auto-coercion of resolver returns (callers' resolvers MUST return a tuple — see task 4).

Update inline doc comments above `normalize_scope` to explain the new contract. Mention that `[scope, nil]` is the canonical "scope-only" tuple shape.

<task type="auto">
  <name>Update with_scope to push tuples and current_scope to read them</name>
  <files>
    lib/typed_eav.rb
  </files>
  <action>
Update `TypedEAV.with_scope(value)`. Keep the public single-arg signature unchanged for BC. Internally, push the value through `normalize_scope` BEFORE pushing on the stack, so the stack always stores `[scope, parent_scope]` tuples (or `nil`).

Change the stack-push site:

  def with_scope(value)
    stack = (Thread.current[THREAD_SCOPE_STACK] ||= [])
    stack.push(normalize_scope(value))
    yield
  ensure
    stack&.pop
  end

This way, `with_scope("t1")` pushes `["t1", nil]`, `with_scope(["t1", "ps1"])` pushes `["t1", "ps1"]`, and the `ensure` pop is unchanged.

Update `current_scope` to read tuples from the stack:

  def current_scope
    return nil if Thread.current[THREAD_UNSCOPED]

    stack = Thread.current[THREAD_SCOPE_STACK]
    return stack.last if stack.present?

    normalize_scope(Config.scope_resolver&.call)
  end

Since the stack already holds normalized tuples (we normalize on push), `stack.last` is returned directly — no second normalize. The resolver-fallback path normalizes the resolver's return value once.

Document on `with_scope` that you may pass a scalar (back-compatible "scope only") or a `[scope, parent_scope]` Array. Keep the old block-API BC pattern obvious in the docstring.
  </action>
