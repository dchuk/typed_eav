  ### Two-level scoping (`parent_scope`)

  When a single tenant axis isn't enough — say, `tenant_id` for the customer
  AND `workspace_id` for an in-tenant partition — declare both:

  ```ruby
  class Project < ApplicationRecord
    has_typed_eav scope_method: :tenant_id, parent_scope_method: :workspace_id
  end
  ```

  Field definitions partition on the tuple `(entity_type, scope, parent_scope)`.
  A `Project` record reads field definitions in three precedence layers:
  full-triple `(scope, parent_scope)` wins, then `(scope, nil)` (tenant-wide),
  then `(nil, nil)` (truly global). Same precedence applies to the query path.

  Both `with_scope` and the configured `scope_resolver` return a tuple now:

  ```ruby
  TypedEAV.with_scope(["t1", "w1"]) do
    Project.where_typed_eav({ name: "status", value: "active" })
  end

  # Single-axis call still works (parent_scope = nil):
  TypedEAV.with_scope("t1") do
    Contact.where_typed_eav({ name: "age", op: :gt, value: 21 })
  end

  # Custom resolver — MUST return [scope, parent_scope]:
  TypedEAV.configure do |c|
    c.scope_resolver = -> { [Current.tenant&.id, Current.workspace&.id] }
  end
  ```

  ### Migrating from v0.1.x

  The resolver-callable contract is a **breaking change**: any custom
  `Config.scope_resolver` lambda must now return `[scope, parent_scope]` (a
  2-element Array) instead of a bare scalar. A scalar return raises
  `ArgumentError` at the next ambient query so the failure is loud, not silent.
  If you don't use parent-scope, return `[scope, nil]`.

  Run `bin/rails typed_eav:install:migrations` to copy the new
  `AddParentScopeToTypedEavPartitions` migration into your app. The migration
  is safe on production: it adds a nullable `parent_scope` column (catalog-only,
  instantaneous) and uses `CREATE INDEX CONCURRENTLY` for all index changes.
  It does NOT touch existing data. Existing fields end up with `parent_scope = NULL`
  (the global-parent shape) and continue to work for every single-scope caller.

  ### Orphan-parent invariant

  A field row with `parent_scope` set and `scope` blank is invalid — model-level
  validation rejects it on save. Reason: a "global field within one workspace"
  makes no semantic sense; the row would never match any record's resolver.
  The paired partial unique indexes rely on this invariant.

In §"Validation Behavior" (around line 470), add one bullet:

  - **Orphan-parent rows rejected**: a `Field` or `Section` with `parent_scope` set but `scope` blank is invalid. The Value-side guard rejects cross-`(scope, parent_scope)` writes too.

In §"Database Support" (line 479), add one sentence at the end of the existing paragraph:

  > As of v0.2.0, the paired partial unique indexes cover the three-key partition tuple `(entity_type, scope, parent_scope)`; the orphan-parent invariant means the `WHERE scope IS NULL` partials don't include `parent_scope`.

Keep prose tight — under ~80 lines added across all README edits. The README is already ~500 lines and densely packed; new content must read like the existing prose.

  ## [0.2.0] - YYYY-MM-DD

  ### Added
  - `parent_scope_method:` kwarg on `has_typed_eav` for two-level partition keys.
  - `parent_scope:` kwarg on `where_typed_eav`, `with_field`, and `typed_eav_definitions`.
  - `TypedEAV.with_scope` accepts a `[scope, parent_scope]` tuple form.

  - **BREAKING** `Config.scope_resolver` callables MUST return a 2-element Array `[scope, parent_scope]`. v0.1.x callables returning a bare scalar will raise `ArgumentError` at the next ambient query.
  - `TypedEAV.current_scope` now returns `[scope, parent_scope]` (or `nil`) — was a String/nil scalar.
  - `Config::DEFAULT_SCOPE_RESOLVER` (acts_as_tenant auto-detect) returns `[ActsAsTenant.current_tenant, nil]`.
  - `Field::Base.for_entity` and `Section.for_entity` accept a `parent_scope:` kwarg (defaults nil).
  - AR uniqueness validators on `Field` (on `:name`) and `Section` (on `:code`) include `parent_scope` in their scope key.

  - Paired partial unique indexes cover the new tuple; old indexes (`idx_te_fields_unique_*`, `idx_te_sections_unique_*`) replaced with `_uniq_` variants.
  - `idx_te_fields_lookup` recreated with `parent_scope` between `scope` and `sort_order`.

  ### Added (validations)
  - `Field::Base#validate_parent_scope_invariant` rejects `parent_scope.present? && scope.blank?`.
  - `Section#validate_parent_scope_invariant` symmetric guard.
  - `Value#validate_field_scope_matches_entity` extended to the parent_scope axis.

  ### Migration steps
  1. Run `bin/rails typed_eav:install:migrations` to copy `AddParentScopeToTypedEavPartitions` into your app.
  2. Run `bin/rails db:migrate`. The migration uses `CREATE INDEX CONCURRENTLY` and is safe on production tables.
  3. Update any custom `TypedEAV.config.scope_resolver` lambda to return `[scope, parent_scope]`. If you don't use parent_scope, return `[scope, nil]`. A bare scalar return surfaces as `ArgumentError` at runtime — there is no silent fallback.
  4. Optional: declare `parent_scope_method:` on hosts that have an in-tenant partition. Existing single-scope models continue to work without changes.

  See the README §"Migrating from v0.1.x" for full guidance.

  [0.2.0]: https://github.com/dchuk/typed_eav/releases/tag/v0.2.0
