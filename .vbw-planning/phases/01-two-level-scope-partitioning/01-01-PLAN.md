<objective>
Add a nullable `parent_scope` (string) column to `typed_eav_fields` and `typed_eav_sections`, drop the five existing scope-partition indexes, and create the new triple-aware paired-partial-unique and lookup indexes for both tables. This is the schema foundation that every other Phase-1 plan depends on.

This is the gem's first follow-on migration after the initial create. It establishes the `disable_ddl_transaction!` + `algorithm: :concurrently` pattern that future production-deployed migrations will follow, since consumer apps may have millions of rows in `typed_eav_fields` / `typed_eav_sections` when they upgrade.

Decision rationale (recorded here for the SUMMARY): include `idx_te_sections_lookup` as a brand-new index for parallelism with `idx_te_fields_lookup` even though Phase 1 success criteria don't strictly require it. RESEARCH §7 flags the asymmetry; adding it now is cheap (one more concurrent CREATE) and avoids a follow-on migration in Phase 2 when section ordering helpers ship.

NULL-handling decision (Option B — paired-partial split): Postgres treats `NULL ≠ NULL` in unique indexes. A naive single partial unique index `(name, entity_type, scope, parent_scope) WHERE scope IS NOT NULL` would NOT prevent two rows with `(name='foo', entity_type='Contact', scope='t1', parent_scope=NULL)` from coexisting — both satisfy `scope IS NOT NULL`, and `parent_scope=NULL` is not equal to itself for uniqueness purposes. We CANNOT rely on the model-level invariant validator (plans 03/04) alone to prevent this: that validator is race-prone under concurrent writes and does not guard direct SQL inserts.

Two options were considered:
- **Option A:** `nulls_not_distinct: true` on a single scoped partial unique index. Requires Postgres ≥ 15 AND a Rails adapter that pipes the option through `add_index` (Rails 7.1+).
- **Option B (chosen):** Split the scoped branch into two partial unique indexes parameterized by whether `parent_scope` is NULL. Always safe across every PG version Rails supports.

Evidence forcing Option B:
- `typed_eav.gemspec:34` — `spec.add_dependency "rails", ">= 7.1"`. Rails 7.1 is the floor.
- `typed_eav.gemspec` and `Gemfile` — no `pg` server-version pin or runtime adapter check.
- `CONCERNS.md` lines 46–55 — declares Postgres-only features (partial unique indexes, NULL-distinct semantics, jsonb @>) but does NOT pin a server-version floor. Consumer apps may be on PG 12/13/14, where `NULLS NOT DISTINCT` does not exist.
- Conclusion: we cannot assume PG ≥ 15. Option B is the only choice that works for every supported deployment.

Trade-off: 3 partial unique indexes per table instead of 2 (6 total scoped/global partials across both tables instead of 4). All names fit comfortably under Postgres' 63-byte identifier limit.

Index name byte counts (all well under 63):
- fields: `idx_te_fields_uniq_scoped_full` (30b), `idx_te_fields_uniq_scoped_only` (30b), `idx_te_fields_uniq_global` (25b), `idx_te_fields_lookup` (20b)
- sections: `idx_te_sections_uniq_scoped_full` (32b), `idx_te_sections_uniq_scoped_only` (32b), `idx_te_sections_uniq_global` (27b), `idx_te_sections_lookup` (22b)
</objective>

Key facts from research:
- Initial migration is the only one today; this establishes follow-on migration conventions.
- Five existing indexes need DROP + CREATE: idx_te_fields_unique_scoped, idx_te_fields_unique_global, idx_te_sections_unique_scoped, idx_te_sections_unique_global, idx_te_fields_lookup. (Pre-existing simple `t.index :entity_type` on fields stays.)
- Paired-partial pattern (Option B, parameterized split): the scoped branch is split into TWO partial unique indexes parameterized by whether `parent_scope` is NULL — `_scoped_full` (scope+parent_scope both set) and `_scoped_only` (scope set, parent_scope NULL). The global partial covers `(cols WITHOUT scope, parent_scope) WHERE scope IS NULL`. Reason: Postgres treats `NULL ≠ NULL` in unique indexes, so a single `(scope, parent_scope) WHERE scope IS NOT NULL` partial would fail to prevent duplicates when `parent_scope` is NULL. The orphan-parent invariant (parent_scope NULL whenever scope NULL) is enforced at the model layer (plans 03/04) and means we don't need a fourth partial for `(parent_scope NOT NULL, scope NULL)` — that combination is rejected before INSERT.
- Index names use `_uniq_` (not `_unique_`) prefix to keep room for the `_full`/`_only` suffixes; longest name `idx_te_sections_uniq_scoped_only` is 33 bytes, well under the 63-byte Postgres identifier limit.

The `_scoped_only` partials are the key bug fix vs. the originally-drafted single-partial design: Postgres unique indexes treat NULLs as distinct, so a single `(scope, parent_scope) WHERE scope IS NOT NULL` index does NOT prevent two rows from coexisting with the same `scope` and `parent_scope = NULL`. Splitting into two partials parameterized on `parent_scope IS NULL` / `IS NOT NULL` closes that hole using only standard Postgres semantics (no `NULLS NOT DISTINCT` required, so works on any PG version Rails 7.1 supports).
