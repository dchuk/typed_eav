<objective>
Deliver the entire cascade refactor as one atomic plan: schema migration (field_dependent column, nullable field_id, FK drop-and-recreate to ON DELETE SET NULL), Field model wiring (validation + before_destroy dispatch, removal of dependent: :destroy), Value model adjustment (belongs_to optional: true), and full spec coverage of all three policies. The migration and the model changes ship together because the model code presumes the new column exists and the FK behavior is changed; splitting them across plans would land a green-tree state where the model expects schema the migration hasn't applied.

This is the only plan that creates the migration in this phase. As wave-1 alone, this plan is the sole writer to field/base.rb during its execution; Plans 02-02 (wave 2) and 02-04 (wave 3) append their additions on top of this plan's merged commit.

The before_destroy dispatch must run BEFORE the field row deletion so the "destroy" policy can call values.destroy_all while the field row is still extant; otherwise the FK ON DELETE SET NULL would fire first and leave orphan rows instead of deleted ones. The "destroy" branch is intentionally inside before_destroy (not after_destroy) to maintain v0.1.0 behavior.
</objective>

Migration body:

```ruby
# frozen_string_literal: true

class AddCascadePolicyToTypedEAVFields < ActiveRecord::Migration[7.1]
  def up
    # Cascade policy column. String (not enum) for forward-compat with future
    # policies; default "destroy" preserves v0.1.0 behavior. NOT NULL because
    # the AR validator narrows to a closed set and the model relies on a
    # non-nil value at every read.
    add_column :typed_eav_fields, :field_dependent, :string,
               null: false, default: "destroy"

    # Allow Value rows to outlive their Field row when field_dependent is
    # "nullify". The existing read-path orphan guards in
    # InstanceMethods#typed_eav_value / typed_eav_hash already skip
    # field-nil rows silently — see CONCERNS.md.
    change_column_null :typed_eav_values, :field_id, true

    # Drop and recreate the FK to switch ON DELETE CASCADE → ON DELETE SET
    # NULL. PG requires drop-and-recreate for an ON DELETE policy change.
    # Using the column-form helpers so we don't hardcode the auto-generated
    # fk_rails_* constraint name.
    remove_foreign_key :typed_eav_values, column: :field_id
    add_foreign_key :typed_eav_values, :typed_eav_fields,
                    column: :field_id, on_delete: :nullify
  end

  def down
    # Reverse FK first (must precede NOT NULL restoration because any orphan
    # rows would otherwise block change_column_null).
    remove_foreign_key :typed_eav_values, column: :field_id
    add_foreign_key :typed_eav_values, :typed_eav_fields,
                    column: :field_id, on_delete: :cascade

    # Re-impose NOT NULL. If any field_id IS NULL rows exist (created while
    # the up-version was live), this raises — operator must clean them up
    # before rolling back. Document in CHANGELOG when shipping.
    change_column_null :typed_eav_values, :field_id, false

    remove_column :typed_eav_fields, :field_dependent
  end
end
```

def dispatch_field_dependent
  case field_dependent
  when "destroy"
    # Explicit destroy_all: with ON DELETE SET NULL on the FK, the DB
    # would otherwise NULL the rows out instead of removing them. AR
    # callbacks run inside the destroy transaction so this is atomic.
    values.destroy_all
  when "nullify"
    # No-op: FK ON DELETE SET NULL nulls field_id automatically. The
    # existing read-path orphan guards (typed_eav_value, typed_eav_hash)
    # silently skip orphan rows, which is the documented fail-soft path.
  when "restrict_with_error"
    return unless values.exists?

    errors.add(:base, "Cannot delete field that has values. Use field_dependent: :nullify or destroy values first.")
    throw(:abort)
  end
end
