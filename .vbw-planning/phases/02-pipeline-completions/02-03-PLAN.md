<objective>
Add the `UNSET_VALUE` sentinel to `TypedEAV::Value` so the non-form `typed_values.create(field: f)` path (no `value:` kwarg) populates the field's configured default. The sentinel is the only mechanism that can distinguish "kwarg not given" from "given as explicit nil" — explicit nil must continue to store nil (not re-apply the default).

The implementation is a minimum-diff sentinel branch in `value=` plus a parallel branch in `apply_pending_value`, mirroring the `UNSET_SCOPE` / `ALL_SCOPES` pattern from `has_typed_eav.rb`. No refactor of the existing pending-value flow is required — RESEARCH §Area 2 confirmed a guard clause is sufficient.

This plan depends on 02-01 (Cascade) because that plan adds `optional: true` to `belongs_to :field` in the same file. Wave-2 placement avoids same-file merge conflict.

The form-construction paths (`typed_values_attributes=`, `typed_eav_attributes=`, `initialize_typed_values`, `set_typed_eav_value`) all pass explicit values and never invoke the sentinel — RESEARCH §Area 2 BC-risk audit confirms this. The new behavior is strictly additive on the non-form path.
</objective>

    # Sentinel branch: caller did NOT pass an explicit value. Apply the
    # field's configured default if field is already assigned; otherwise

    # Rails will further cast via the column type on save.

**Edit 4 — override initialize so missing `value:` kwarg becomes UNSET_VALUE.**

This is the wiring that makes `typed_values.create(field: f)` (no `value:` kwarg) actually invoke the sentinel branch. Rails' AR doesn't intercept missing kwargs natively — without this override, `create(field: f)` would just leave the typed column nil (the existing v0.1.0 behavior). Detecting the "kwarg was given as nil" vs "kwarg not given" distinction is impossible after the fact (both produce the same nil typed column), so the substitution must happen at construction time, BEFORE `super` calls into AR's `assign_attributes`.

Insert as a public method just BEFORE the `# ── Callbacks ──` comment / `after_initialize :apply_pending_value` declaration (current line 58). It must run before `assign_attributes` so the substituted UNSET_VALUE makes it into the `value=` setter path.

```ruby
# Override AR's initialize so missing `:value` kwarg → UNSET_VALUE
# substitution. This is the only mechanism that lets us distinguish
# "no value given" from "value: nil given" (both leave the typed column
# nil; the difference can only be observed at construction time). The
# sentinel then flows through `value=` and (if field is unset) into
# `@pending_value`, where `apply_pending_value` resolves it to the
# field's configured default once field becomes available.
#
# `accepts_nested_attributes_for` paths and `set_typed_eav_value` always
# pass an explicit `value:` (never missing the key), so they bypass this
# substitution and continue to behave as before.
def initialize(attributes = nil, &block)
  if attributes.is_a?(Hash)
    attrs = attributes.dup
    attrs[:value] = UNSET_VALUE unless attrs.key?(:value) || attrs.key?("value")
    super(attrs, &block)
  elsif defined?(ActionController::Parameters) && attributes.is_a?(ActionController::Parameters)
    # Permitted params hash-like: convert to a plain hash for the key check,
    # then re-pass. Same UNSET_VALUE substitution rule.
    attrs = attributes.to_h
    attrs[:value] = UNSET_VALUE unless attrs.key?(:value) || attrs.key?("value")
    super(attrs, &block)
  else
    # nil, scalar, or any other shape AR's initialize accepts unchanged.
    super
  end
end
```

**Late-field-assignment caveat (documented behavior):** The contract is "default population fires on the create path." If a caller does `v = Value.new(entity: c); v.field = f; v.save`, the sentinel was never invoked (no value: kwarg was missing — there was no value: kwarg at all in the second pattern), so no default fires. Callers in this rare pattern can opt in by explicitly setting `v.value = TypedEAV::Value::UNSET_VALUE` after assigning the field, OR by passing `value: TypedEAV::Value::UNSET_VALUE` in the constructor. This is consistent with the CONTEXT.md scope ("non-form construction path") — late `.field=` assignment is not the supported create path.
