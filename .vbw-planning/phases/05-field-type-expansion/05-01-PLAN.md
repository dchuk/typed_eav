<objective>
Land the three Field::Base extension points that all five Phase 5 field types depend on, with all wiring at consumer sites (Value, QueryBuilder) preserving BC for the 17 existing types. After this plan: external field types (and the Phase 5 Currency / Reference classes that follow) can override `read_value`, `apply_default_to`, and `operator_column` to compose multi-cell logical values and route operators to different physical columns. NO new field types are added in this plan — that's intentional. The infrastructure must be proven BC-safe in isolation before Currency / Reference / Image / File / Percentage exercise it.

This plan ships ONLY the dispatch infrastructure. The five new field classes, their factories, their value-spec column assertions, their `BUILTIN_FIELD_TYPES` registrations, and the `:references` / `:currency_eq` operator implementations all land in plans 05-02 (Currency + Percentage), 05-03 (Image + File), and 05-04 (Reference). The split is along file-conflict boundaries: this plan is the only one touching `app/models/typed_eav/field/base.rb`, `lib/typed_eav/column_mapping.rb`, the read-side of `app/models/typed_eav/value.rb`, and the column-selection line of `lib/typed_eav/query_builder.rb`. Each downstream plan touches its own field class file plus a single-line `BUILTIN_FIELD_TYPES` insertion in `lib/typed_eav/config.rb`.

After this plan ships and migrations apply, calling `Value#value` and `Value#apply_field_default` is observably identical for every existing field type — the dispatch overhead is a single method call. `QueryBuilder.filter` returns the same Arel as before for every existing operator on every existing type. The three new methods exist with their default implementations, are publicly callable, are unit-spec covered, and are documented in README. Plan 05-02 / 05-03 / 05-04 only need to override these methods inside their own field class files — no further changes to Field::Base, ColumnMapping, Value (read side), or QueryBuilder are required.
</objective>

#
# These two instance methods are the field-side surface that resolves
# Value#value semantics and the default-application path. Single-cell
# field types (every built-in as of Phase 4) inherit the defaults below
# and behave identically to the pre-Phase-5 direct-column-access shape.

# Value#value and Value#apply_field_default oblivious to multi-cell — they

# record. Default reads `value_record[self.class.value_column]`. Override
# in multi-cell field types to compose a hash from multiple columns
# (e.g., Field::Currency returns `{amount: r[:decimal_value], currency: r[:string_value]}`).
#
# Called from Value#value. The Value#value `return nil unless field`
# guard runs before this method, so `self` is always set.
def read_value(value_record)

# Default writes `value_record[self.class.value_column] = default_value`,
# bypassing Value#value= to avoid re-casting an already-cast default
# (default_value is cast at field save time via validate_default_value).

# composite default (e.g., Field::Currency unpacks `default_value`'s

#   2. Pending-value resolution (apply_pending_value branch where

def apply_default_to(value_record)
  value_record[self.class.value_column] = default_value
end
