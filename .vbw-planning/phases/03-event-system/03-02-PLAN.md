Pre-Lead resolutions binding on this plan:
- `on_field_change` arity is `(field, change_type)` — TWO args, no context. EventDispatcher.dispatch_field_change passes only those two. Specs must assert this (a spec that passes 3 args to the user proc would break the locked contract).
- `current_context` returns frozen `{}` when stack empty (delivered by plan 03-01); user procs receive a frozen Hash and mutation must raise FrozenError — which is then caught by the user-proc rescue. This is intentional: a buggy proc that tries `ctx[:foo] = bar` will be logged-and-swallowed rather than crashing the save.
- Reset split: `Config.reset!` resets user procs; `EventDispatcher.reset!` resets ONLY internal-subscribers arrays. `:event_callbacks` opt-in metadata uses snapshot/restore — NOT reset! — so engine-load registrations from Phase 04 versioning (when it lands) survive across examples.
- Use `after_commit :method, on: :create` form (NOT `after_create_commit :method`) for all Value callbacks. Rails 8.1 has a documented alias-collision bug where `after_create_commit :foo` + `after_update_commit :foo` causes only the last to fire. Scout verified via context7 (`/rails/rails/v8.1.2`).
- Field::Base uses a SINGLE `after_commit :_dispatch_field_change` (no `on:` filter) with internal branching. Reason: `:rename` requires `saved_change_to_attribute?(:name)` and lives inside the `:update` lifecycle; one-method branching is cleaner than splitting `:rename` into the `on: :update` callback.

Newly-created-record idiom: use `created?` (alias of `previously_new_record?`, available since Rails 6.1; Gemfile.lock pins activerecord 8.1.3). NOT `id_previously_changed?` — that works but is incidental (relies on PK going nil → assigned during INSERT). `created?` is intent-revealing and the documented Rails 8 form. Scout's research §B uses `created?` — follow that recommendation.

Foundational principle: NO HARDCODED ATTRIBUTE REFERENCES. Value's `:update` filter uses `field.class.value_column` (reads the typed column name from the field's class). Field's `:rename` branch uses `saved_change_to_attribute?(:name)` — the symbol `:name` is structural, not an attribute-value access; it's the locked detection mechanism per 03-CONTEXT.md (and is the one Field column whose change is semantically a separate event type).

Phase 02 cascade interaction (binding context — verified by Scout §B against 02-01-SUMMARY.md):
- `field_dependent: "destroy"` → `values.destroy_all` runs in Field#before_destroy → Value `after_commit on: :destroy` fires per Value row → Field `after_commit on: :destroy` fires once. All `after_commit` callbacks fire after the OUTERMOST commit (single transaction), so Value :destroy events precede Field :destroy event in the same post-commit batch.

- `field_dependent: "restrict_with_error"` → throw(:abort) prevents Field destroy. NO events fire.

# field_id NULLed by the Phase 02 ON DELETE SET NULL FK when a Field
# with field_dependent: :nullify was destroyed). The event contract

# pile up no-op version rows (every audit-trail commit) and Phase 07
# matview would refresh on bookkeeping-only writes.

def _dispatch_value_change_create
  return unless field
  TypedEAV::EventDispatcher.dispatch_value_change(self, :create)
end

def _dispatch_value_change_update
  return unless field
  return unless saved_change_to_attribute?(field.class.value_column)

  TypedEAV::EventDispatcher.dispatch_value_change(self, :update)
end

def _dispatch_value_change_destroy
  return unless field

  TypedEAV::EventDispatcher.dispatch_value_change(self, :destroy)
end
```

```ruby
# ── Phase 03 event dispatch ──
#
# Branches the four change_types via lifecycle predicates in the order
# locked by 03-CONTEXT.md §"`on_field_change` change_type set" → "Mechanism":
#
#   created? → :create
#   destroyed? → :destroy
#   saved_change_to_attribute?(:name) → :rename
#   else → :update
#
# The four predicates are mutually exclusive per save (`created?` and
# `destroyed?` cannot both be true; rename/update only reachable when
# neither create nor destroy fired), so the ordering is behaviorally
# equivalent to any other ordering — but matching CONTEXT.md verbatim
# keeps the code grep-able against the locked decision and avoids the
# next reader wondering whether the order was intentional or accidental.
#
# `created?` is the Rails 6.1+ alias of `previously_new_record?` and is
# the intent-revealing form per Scout RESEARCH §B (vs. id_previously_changed?
# which works but reads as accidental — relies on PK going nil → assigned
# during INSERT).
#
# Rename detection is structural: any save where the :name column changed
# counts as a rename, even if combined with other attribute changes
# (sort_order, options, default_value, field_dependent). This false-
# positive bias is intentional — Phase 07's matview must regenerate
# column DDL on rename; missing a rename combined with other edits would
# corrupt the matview's column-name → field-name map.
#
# `:name` is the only attribute name we hardcode in this callback, and
# it's structural (the locked rename-detection mechanism per 03-CONTEXT.md),
# not a data access — we don't read the value of `name`, only that it
# changed.
def _dispatch_field_change
  change_type = if created?
                  :create
                elsif destroyed?
                  :destroy
                elsif saved_change_to_attribute?(:name)
                  :rename
                else
                  :update
                end
  TypedEAV::EventDispatcher.dispatch_field_change(self, change_type)
end
```

They are orthogonal: a unit spec exercising `EventDispatcher` directly with doubles needs `:event_callbacks` for state isolation but does NOT need real commits (no AR records). An integration spec creating real Field/Value rows needs `:real_commits` so callbacks fire durably AND `:event_callbacks` so dispatch state is isolated. # example starts clean, and restore on exit." Uses snapshot-and-restore
# rather than EventDispatcher.reset! / Config.reset! because Phase 04
# versioning will register internal subscribers at engine load — wiping

config.around(:each, :event_callbacks) do |example|

## Event hooks

`typed_eav` fires `after_commit` events for value and field changes. Use them
for audit logs, search-index synchronization, cache invalidation, or any
out-of-band reaction that must wait until the database write is durable.

### Public callback slots

```ruby
TypedEAV.configure do |c|
  c.on_value_change = ->(value, change_type, context) {
    # change_type ∈ [:create, :update, :destroy]
    # context is a frozen Hash (see `with_context` below) — read-only
  }

  c.on_field_change = ->(field, change_type) {
    # change_type ∈ [:create, :update, :destroy, :rename]
    # NOTE: no context arg — field changes are CRUD-on-config, not
    # per-entity user actions
  }
end
```

The `:rename` change_type fires whenever the field's `name` column changed
in the just-committed save, even when bundled with other attribute changes
(options, sort_order, default_value, etc.). The detection is intentionally
escalating — Phase 7's materialized index needs to regenerate column DDL on
every rename.

`:update` on Value fires only when the typed value column changed. Saving
a Value record without modifying its typed column (e.g., touching only
bookkeeping columns) is a no-op for event dispatch.

`field_dependent: :nullify` cascades produce **no** Value `:destroy`
events. The FK `ON DELETE SET NULL` runs at the database level and
bypasses AR callbacks. Only the Field `:destroy` event fires. Use
`field_dependent: :destroy` if your consumer needs per-Value events on
field deletion.

### Thread-local context with `with_context`

```ruby
TypedEAV.with_context(request_id: request.uuid, actor_id: current_user.id) do
  contact.update!(typed_eav: { phone: "555-1234" })
  # on_value_change receives { request_id: "...", actor_id: 42 } as context
end
```

`with_context` is a thread-local stack with shallow per-key merge:

```ruby
TypedEAV.with_context(request_id: "abc") do
  TypedEAV.with_context(source: :bulk) do
    # current context: { request_id: "abc", source: :bulk }
  end
  # current context: { request_id: "abc" }
end
# current context: {}
```

The current-context hash is frozen — callbacks cannot mutate it. Outer
context is restored on exit even if the inner block raises.

`TypedEAV.current_context` returns the current frozen Hash (or a shared
frozen `{}` when no `with_context` block is active). It's safe to call
from any code path; it never returns nil.

### Error policy

User callbacks (`Config.on_value_change`, `Config.on_field_change`) are
rescued — exceptions are logged via `Rails.logger.error` and **do not
propagate** to the user's save call. The save row is already committed
when `after_commit` fires; re-raising would surface a misleading
"save failed" error.

This is the deliberate split with first-party features. Internal
subscribers used by `typed_eav` itself (Phase 4 versioning, Phase 7
materialized index) follow a different rule: their exceptions
**propagate**. Versioning corruption must be loud.

### Ordering guarantee

When multiple subscribers are registered, they fire in this order:

1. First-party internal subscribers (versioning, matview, etc.), in
   registration order. Errors propagate.
2. The user proc on `Config.on_value_change` / `Config.on_field_change`,
   last. Errors are rescued and logged.

Reassigning `Config.on_value_change` after gem initialization does **not**
disable internal subscribers — they live on a separate dispatcher list
and survive `Config.reset!`.

### Test isolation

Test files that exercise event hooks should opt in to the `:event_callbacks`
metadata:

```ruby
RSpec.describe "my feature", :event_callbacks do
  it "fires the hook" do
    captured = []
    TypedEAV::Config.on_value_change = ->(v, t, _ctx) { captured << [v.id, t] }
    contact.update!(typed_eav: { phone: "555-1234" })
    expect(captured).to include([be_a(Integer), :update])
  end
end
```

The `:event_callbacks` around hook in `spec/spec_helper.rb` snapshots and
restores Config user procs and the internal-subscriber lists around each
example, so test mutations don't leak across examples and engine-load
registrations from later phases stay intact.
