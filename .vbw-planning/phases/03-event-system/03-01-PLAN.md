Key locked decisions binding on this plan (from 03-CONTEXT.md "Decisions Made"):
- `Config.on_value_change` / `Config.on_field_change` are PUBLIC single-proc slots backed by ActiveSupport::Configurable.
- A PRIVATE internal-subscribers list lives on EventDispatcher (NOT on Config). Internals fire FIRST in registration order; user proc fires LAST.
- Internal subscribers raise (fail-closed — versioning corruption must be loud). User proc is rescue StandardError + Rails.logger.error + no re-raise.
- `with_context(**kwargs)` is a thread-local stack with shallow per-key merge and ensure-pop.
- `on_field_change(field, change_type)` deliberately takes 2 args (no context). Pre-Lead resolution: this asymmetry vs `on_value_change(value, change_type, context)` is intentional and locked — do NOT add context.
- EventDispatcher is eager-loaded (NOT autoloaded) — Phase 04 versioning will register at engine boot.
- `current_context` returns `{}.freeze` (frozen) when stack is empty, NOT bare `{}`.
- Reset split: `Config.reset!` resets user procs; `EventDispatcher.reset!` resets ONLY internal-subscribers arrays. Does NOT touch Config user procs (Phase 04/07 engine-load registrations live on the dispatcher and must survive Config.reset!).
- Use `after_commit :method, on: :create` form throughout (alias-collision bug). N/A in this plan but stays binding for plan 02.

Pattern conformance:
- ActiveSupport::Configurable for Config (PATTERNS.md §"Configurables via ActiveSupport::Configurable")
- private_constant for thread-local keys (PATTERNS.md §"Thread-locals namespaced and private_constant")
- `frozen_string_literal: true` on every new .rb (CONVENTIONS line 87)
- "Comment the failure mode" — every new comment explains what would break if removed (PATTERNS.md §"Comment the failure mode")
- "Errors tell you how to fix it" — error/log messages include enough context to debug inline (PATTERNS.md)

Foundational principle: NO HARDCODED ATTRIBUTE REFERENCES. The dispatch methods receive Value/Field objects as parameters — never bake in attribute names. (Plan 02 will use `field.class.value_column` and `saved_change_to_attribute?(:name)` — the symbol `:name` is acceptable per locked decision since `:rename` detection is structural, not a hardcoded attribute access.)

3. `def with_context(**kwargs)`:
   - Body mirrors `with_scope` (lines 124-130): `stack = (Thread.current[THREAD_CONTEXT_STACK] ||= [])`.
   - Compute `merged = (stack.last || {}).merge(kwargs).freeze` — outer-stack + new kwargs, frozen so callbacks cannot mutate.
   - `stack.push(merged)`.
   - `yield`.
   - `ensure; stack&.pop`.
   - Doc comment block above the method explains: "Thread-local context stack with shallow per-key merge. Mirrors with_scope's ensure-pop semantics. Pushes a frozen Hash so callback mutation raises FrozenError (caught by EventDispatcher's user-proc rescue). Use **kwargs (not positional Hash) to enforce keyword-syntax — `TypedEAV.with_context({ foo: 1 })` raises ArgumentError in Ruby 3.0+ kwargs/Hash separation; `TypedEAV.with_context(foo: 1)` is the only accepted form. Without freeze, a callback that did `ctx[:added] = true` would corrupt the stack for the current and outer with_context blocks. Without **kwargs, callers could push arbitrary Hash shapes that wouldn't merge cleanly across nesting."

4. `def current_context`:
   - Body: `Thread.current[THREAD_CONTEXT_STACK]&.last || EMPTY_FROZEN_CONTEXT`
   - Define `EMPTY_FROZEN_CONTEXT = {}.freeze` as a private_constant immediately above `THREAD_CONTEXT_STACK` (or as a `class << self`-local `EMPTY_FROZEN_CONTEXT = {}.freeze` constant — pick whichever is more idiomatic; the latter is preferred since `current_context` is the only caller). Comment: "Returning a single shared frozen instance avoids allocating a new Hash on every empty-context dispatch (the hot path inside `dispatch_value_change` when no with_context block is active). Per locked decision (03-CONTEXT.md, pre-Lead resolution): always-frozen, never bare {}."
   - Doc comment above the method: "Returns the current thread's top-of-stack context Hash, or a shared frozen empty Hash when no with_context block is active. The return value is ALWAYS frozen — callers can rely on read-only semantics regardless of whether a block is active."

```ruby
# Public single-proc slot for value-change events.
# Signature: ->(value, change_type, context) { ... }
# - value:        TypedEAV::Value (the just-committed row)
# - change_type:  :create | :update | :destroy
# - context:      Hash (TypedEAV.current_context — frozen)
#
# Errors raised inside this proc are rescued by EventDispatcher and
# logged via Rails.logger.error — they do NOT propagate to the
# user's save call (the row is already committed). Internal subscribers
# (Phase 04 versioning, Phase 07 matview) fire BEFORE this proc and
# their errors DO propagate. See 03-CONTEXT.md §User-callback error policy.
#
# Reassignment after gem initialization does NOT disable internal
# subscribers — those live on EventDispatcher.value_change_internals,
# not here.
config_accessor :on_value_change, default: nil

# Public single-proc slot for field-change events.
# Signature: ->(field, change_type) { ... }
# - field:        TypedEAV::Field::Base (or subclass)
# - change_type:  :create | :update | :destroy | :rename
#
# Note: TWO args, no context — asymmetric vs on_value_change by design.
# Field changes are CRUD-on-config (admin operations on field
# definitions), not per-entity user actions, so thread context is
# less relevant. The asymmetry is locked at 03-CONTEXT.md §Phase Boundary.
#
# :rename fires when `name` is among Field#saved_changes, even
# combined with other attr changes (sort_order, options, etc.) —
# Phase 07 matview needs the rename signal to regenerate column names
# even when the rename was bundled with other edits.
config_accessor :on_field_change, default: nil
```
