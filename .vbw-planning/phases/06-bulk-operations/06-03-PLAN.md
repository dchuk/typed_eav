<objective>
Land `TypedEAV::CSVMapper.row_to_attributes(row, mapping, fields_by_name:)` as a pure stateless module method that transforms a single CSV row into an attributes Hash suitable for `typed_eav_attributes=` consumption. After this plan: callers pass a `CSV::Row` (or `Array`), a mapping (header→field-name OR index→field-name), and `fields_by_name` (resolved from the host record's partition); receive back a `Result` with `.attributes`, `.errors`, `.success?`. Type coercion goes through existing `Field#cast`. Per-row errors land in `errors`, never raise.

This plan is INDEPENDENT of bulk write (Plan 04), bulk read (Plan 05), schema export/import (Plan 02), and the version_group_id migration (Plan 01). It creates two new files and adds one line to `lib/typed_eav.rb`. No file conflict with any other Phase 6 plan — runs in wave 1 alongside Plan 01 and Plan 02.

Why isolated: the CSV mapper is a pure functional transform with no AR dependency beyond `field.cast`. It does not read or write the database. It does not depend on bulk write semantics or versioning. The autoload registration in `lib/typed_eav.rb` is one new line in a section of the file untouched by any other Phase 6 plan.
</objective>

Locked decisions binding on this plan (from 06-CONTEXT.md):

- **Mapping shape:** Single Hash. String keys → CSV header names; Integer keys → column indexes. No transform overrides in v0.6.0.
- **Per-row return:** `TypedEAV::CSVMapper::Result` value with `.attributes`, `.errors`, `.success?`. Never raises on row-level errors.
- **Streaming:** Per-row API only. No file-level orchestration. Callers use `CSV.foreach(path, headers: true) { |row| … }`.
- **Coercion:** Existing `Field#cast` tuple contract. Cast failures land in `errors`, NOT exceptions.

Plan-time resolutions (RESEARCH §Open-Question Resolutions):

- **Result#errors shape:** `{ \"field_name\" => [\"error message\"] }` plain Hash with String field-name keys. Matches AR's `errors.messages` shape but using string keys to match the existing `field.name` convention.

Anchors confirmed in research:
- No existing CSV code in the gem — entirely new module (RESEARCH §CSV mapper Anchor 1).
- File placement: `lib/typed_eav/csv_mapper.rb`. Autoload registration: `lib/typed_eav.rb` line 14–21 alphabetical block (RESEARCH §CSV mapper Anchor 4).
- `field.cast(raw)` returns `[casted_value, invalid?]` (RESEARCH §CSV mapper Anchor 2).
- Error shape symmetric with `bulk_set_typed_eav_values#errors_by_record` (Plan 04) — both use `{ field_name => [messages] }` (RESEARCH §CSV mapper Anchor 3).

⚠ Discrepancy awareness:
- The `CSV` stdlib has TWO row representations: `CSV::Row` (when `headers: true`) and `Array` (when `headers: false`). Tests must cover BOTH paths.
- `CSV::Row#[]` accepts string AND symbol keys; we standardize on string keys per the codebase convention. The mapper passes mapping keys VERBATIM to `row[key]` — so a mapping of `{\"Email\" => :email}` reads `row[\"Email\"]`, while `{0 => :email}` reads `row[0]`.
- The mapping VALUES (the field names) are accepted as either symbols or strings — the mapper coerces to string before lookup in `fields_by_name`. This matches the codebase convention (`field.name` is always a String).

<notes>
Do NOT introduce per-row exception rescuing. The contract is "field.cast never raises on row-level cell content"; if `field.cast` somehow raises (a bug in a custom field type), the mapper SHOULD propagate the exception — that's a system-level error, not a row-level coercion failure. Don't swallow it with a blanket `rescue StandardError`.

The `Result` value object intentionally has NO mutation API. Callers that need to combine multiple row results into a batch view do so by composing the immutable Hashes in their own code (e.g., `results.flat_map(&:errors).reduce({}, :merge)`). The mapper does not provide a "merge" helper in v0.6.0.

For the mixed-key check: `mapping.keys.map(&:class).uniq` — if the result includes BOTH String and Integer (or any non-String, non-Integer class), raise. Allow homogeneous String OR homogeneous Integer keys.
</notes>
