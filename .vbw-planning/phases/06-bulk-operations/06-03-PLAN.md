<objective>
Land `TypedEAV::CSVMapper.row_to_attributes(row, mapping, fields_by_name: nil)` as a pure stateless module method that transforms a single CSV row into an attributes Hash suitable for `typed_eav_attributes=` consumption. The public 2-arg signature `row_to_attributes(row, mapping)` matches CONTEXT line 13 and ROADMAP §Phase 6 success criterion exactly; the optional 3rd `fields_by_name:` keyword is the typed-coercion path. After this plan: callers pass a `CSV::Row` (or `Array`), a mapping (header→field-name OR index→field-name), and OPTIONALLY `fields_by_name` (resolved from the host record's partition); receive back a `Result` with `.attributes`, `.errors`, `.success?`. Two operating modes:
- **Typed mode** (`fields_by_name:` supplied): per-cell coercion through existing `Field#cast`; cast failures land in `errors`. Never raises on per-row errors.
- **Passthrough mode** (`fields_by_name:` omitted/nil): raw cell values keyed by mapped field name flow through unchanged; `errors` is always empty (no coercion attempted); `success?` is true unless the mapping itself was malformed (mixed-key ArgumentError still raises in either mode).

This plan is INDEPENDENT of bulk write (Plan 04), bulk read (Plan 05), schema export/import (Plan 02), and the version_group_id migration (Plan 01). It creates two new files and adds one line to `lib/typed_eav.rb`. No file conflict with any other Phase 6 plan — runs in wave 1 alongside Plan 01 and Plan 02.

Why isolated: the CSV mapper is a pure functional transform with no AR dependency beyond `field.cast`. It does not read or write the database. It does not depend on bulk write semantics or versioning. The autoload registration in `lib/typed_eav.rb` is one new line in a section of the file untouched by any other Phase 6 plan.
</objective>

Locked decisions binding on this plan (from 06-CONTEXT.md):

- **Mapping shape:** Single Hash. String keys → CSV header names; Integer keys → column indexes. No transform overrides in v0.6.0.
- **Per-row return:** `TypedEAV::CSVMapper::Result` value with `.attributes`, `.errors`, `.success?`. Never raises on row-level errors.

- **Coercion:** Existing `Field#cast` tuple contract. Cast failures land in `errors`, NOT exceptions.

Plan-time resolutions (RESEARCH §Open-Question Resolutions):

- **Result#errors shape:** `{ \"field_name\" => [\"error message\"] }` plain Hash with String field-name keys. Matches AR's `errors.messages` shape but using string keys to match the existing `field.name` convention.

⚠ Discrepancy awareness:
- The `CSV` stdlib has TWO row representations: `CSV::Row` (when `headers: true`) and `Array` (when `headers: false`). Tests must cover BOTH paths.
- `CSV::Row#[]` accepts string AND symbol keys; we standardize on string keys per the codebase convention. The mapper passes mapping keys VERBATIM to `row[key]` — so a mapping of `{\"Email\" => :email}` reads `row[\"Email\"]`, while `{0 => :email}` reads `row[0]`.

## Plan Amendments

### DEV-02: csv runtime dependency added (typed_eav.gemspec)

**Classification:** plan-amendment (assumption invalidated by execution environment; surgical additive fix).

**Original plan-time intent:** The plan author asserted that `require "csv"` would resolve from Ruby's stdlib without any gemspec change, on the grounds that `csv` had been a default gem since Ruby 3.0. The plan's must_haves block included the line: "The `csv` stdlib is always available in Ruby ≥ 3.0; no gemspec change. Loading is lazy via the autoload registration."

**Reality at execution:** Ruby 3.4 removed `csv` from the default gems list (it's now a bundled gem, available on disk but not auto-loaded under bundler). The dev environment runs Ruby 3.4.4. Without an explicit gemspec dependency, `bundle exec rspec spec/lib/typed_eav/csv_mapper_spec.rb` fails immediately at `require "csv"` with `LoadError: cannot load such file -- csv`. The plan author's assumption was correct at plan-write time (Ruby ≤ 3.3) but did not hold for the deployed Ruby version.

**Fix shipped:** One surgical additive line in `typed_eav.gemspec`:
```
spec.add_dependency 'csv', '~> 3.3'
```
The `~> 3.3` constraint matches the version Rails 8.x already pulls in transitively, keeping the dependency window narrow. `required_ruby_version` is unchanged (`>= 3.1`) — the gem still supports Ruby 3.1, 3.2, 3.3 (where csv is a default gem and the dependency is harmless) AND Ruby 3.4+ (where the explicit dependency is now required).

**Sequencing:** The gemspec change was committed BEFORE the implementation commit so the new module's `require "csv"` resolves cleanly at autoload time. Commit order:
- `f03311b` chore(deps): declare csv as runtime dependency  ← prereq
