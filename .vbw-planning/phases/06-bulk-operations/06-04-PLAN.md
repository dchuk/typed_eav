Locked decisions binding on this plan (from 06-CONTEXT.md):

- **Bulk read API:** `Entity.typed_eav_hash_for(records)` returns `{ entity_id => { name => value } }`. Internally: one preload of `typed_values: :field`, group by entity, apply `definitions_by_name` collision precedence.
- **Phase 7 cache integration deferred:** ship preload-only here. Phase 7 will add cache hooks later without changing this method's signature.

Anchors confirmed in research:
- The instance method `typed_eav_hash` (has_typed_eav.rb:584–606) is the per-record reference implementation. The class method is the bulk version preloaded across many records.
- `loaded_typed_values_with_fields` (private instance method at 613–623) is what the instance method uses; the class method's bulk preload is the equivalent at scale: `Value.includes(:field).where(entity: records)`.
- `HasTypedEAV.definitions_by_name(defs)` (lines 57–61) is the SHARED collision-precedence function. Use it. Do NOT inline the sort/index_by logic.
- `typed_eav_definitions(scope:, parent_scope:)` is the per-tuple field lookup. Group records by partition tuple FIRST, then call once per tuple.
- The `typed_values` association is polymorphic (`as: :entity`) declared at lines 134–139.

⚠ Discrepancy awareness:
- Records can have `nil` scope and/or `nil` parent_scope. The grouping must treat `[nil, nil]` as a valid tuple (the global partition). Use `records.group_by { |r| [r.typed_eav_scope, r.typed_eav_parent_scope] }` directly — Ruby Hash handles nil keys cleanly.
- For unscoped host models (e.g., Product without `scope_method:`), `record.typed_eav_scope` returns nil unconditionally — the macro defines the method to return nil when `*_method` is not declared (CONVENTIONS.md §Naming line 118).
- The `definitions_by_name` collision-precedence already handles "global + scoped name collision wins to scoped" — reusing it ensures the bulk-read result is consistent with what the instance method `typed_eav_hash` would return.
- The `Value.includes(:field).where(...)` query relies on the polymorphic association — no manual `entity_type:` filter needed when using `where(entity: records)`. BUT: `where(entity: records)` requires all records to be of a single class for the polymorphic shortcut. Per the validation in this plan (single-class records only), this is fine. Alternatively, use the explicit form `where(entity_type: name, entity_id: records.map(&:id))` for clarity — both are equivalent for single-class input.

<notes>
The single-shot value preload across ALL records (not per-tuple) is intentional: SQL `WHERE entity_type = ? AND entity_id IN (?)` is a single index seek regardless of how many partition tuples are represented. Splitting per-tuple would issue N queries instead of 1.

The PER-tuple `winning_ids_by_name` map IS necessary because field definitions vary by partition. A field name "age" in tenant_1 may have a different field_id than "age" in tenant_2; the collision-precedence map must be computed per-tuple to apply correctly.

Do NOT pre-compute `definitions_by_name` outside the iteration — defer to the existing `HasTypedEAV.definitions_by_name` shared function. This guarantees parity with the instance-method path.

For STI host models: `is_a?(self)` accepts subclasses (e.g., a `PremiumContact < Contact` record passes `is_a?(Contact)`). The class-name in the value query (`entity_type: name`) uses the parent class name — Phase 1 partition tuple is keyed off the registered `entity_type`, NOT the runtime class. This matches existing Phase 1 STI behavior; no special handling needed.
</notes>
