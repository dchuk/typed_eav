⚠ Discrepancy awareness:
- The `after_commit` hooks fire on the OUTER transaction commit, NOT on savepoint release. The `with_context(version_group_id: uuid)` block must be ACTIVE at the time the after_commit dispatch happens — which means the outer transaction must commit while the with_context is still active. This means: `with_context { transaction { records.each { transaction(requires_new: true) { record.save } } } }` — the with_context wraps the OUTER transaction, NOT each savepoint. (RESEARCH §Risks §Savepoint-in-after_commit interaction.)
- For `:per_record`: the `with_context` is per-record-but-still-outside-the-savepoint. Specifically: `transaction { records.each { with_context(version_group_id: SecureRandom.uuid) { transaction(requires_new: true) { record.save } } } }`. The with_context spans the savepoint-and-its-after_commit-deferral; multiple records each get their own UUID active in turn.
- For `:per_field`: the savepoint per record requires an inner save per field (one save per field-name, each wrapped in its own `with_context`). This is slower than the single `record.save` of `:per_record`; the spec verifies the version rows show the correct grouping; the implementation comment explains the tradeoff.
- `typed_eav_attributes=` calls `typed_values.build` for new values OR `typed_values_attributes=` for existing values (RESEARCH §Bulk write Anchor 1). For `:per_field`, splitting the save into one-save-per-field means each save flushes one Value at a time — the existing `accepts_nested_attributes_for :typed_values` machinery handles this correctly. Document this in the inline comment.
- The outer transaction's `after_commit` fires once for ALL successful records together. The Phase 4 versioning subscriber processes each Value change individually — one version row per Value change. The `version_group_id` for each version row reflects the with_context active at the time `EventDispatcher.dispatch_value_change` was called, which is per-record (or per-field) inside the loop.
- A record that fails validation rolls back its SAVEPOINT — its `after_commit` does NOT fire (the savepoint rollback voids the staged callbacks). This is correct: failed records produce no version rows.
- The `errors_by_record` keys: use `record` (the AR object itself) — NOT `record.id`, because records that have not been saved yet (or that fail BEFORE getting an id) would have nil ids. Document this in the inline comment.

<notes>
This task is the architectural inflection point of Phase 6. The savepoint-per-record vs version_grouping tension was identified in RESEARCH §Risks §Savepoint-in-after_commit interaction and CONTEXT §Bulk write failure isolation but not fully resolved. The Lead resolves it here at plan time per the protocol "settled in planning without user input" approach in CONTEXT.

The deliberate design decision: when `version_grouping == nil` (default), preserve outer-transaction + savepoint semantics (cross-record atomicity). When version_grouping is requested, switch to per-record top-level transactions (relaxed atomicity, but version_group_id correctly populated).

Specs MUST cover both modes:
- `version_grouping: nil` + outer transaction failure (e.g., simulated connection drop): all records roll back.
- `version_grouping: :per_record` + one record fails: that record's transaction rolls back; OTHER records still commit (because they ran in independent transactions). The error_by_record shape captures the failure.

This is the v0.6.0 contract; document in README §"Bulk Operations" and CHANGELOG §Phase 6 entry that strict outer atomicity with version grouping is a future enhancement.
