<objective>
Land `TypedEAV::Field::Base.export_schema` and `TypedEAV::Field::Base.import_schema` as class methods on `Field::Base`. After this plan: `Field::Base.export_schema(entity_type: "Contact", scope: "tenant_1")` returns a serializable Hash containing all field rows, all section rows, all `field_options` rows for select/multi_select fields, and the `schema_version: 1` envelope; `Field::Base.import_schema(hash, on_conflict: :error|:skip|:overwrite)` round-trips that hash back into the database idempotently. Type-swap is always raised. Schema-version mismatch is always raised.

This plan is INDEPENDENT of bulk write (Plan 03), bulk read (Plan 04), CSV mapper (Plan 05), and the version_group_id migration (Plan 01). It modifies `app/models/typed_eav/field/base.rb` plus a new spec file. No file conflict with any other Phase 6 plan — runs in wave 1 alongside Plan 01 and Plan 05.

Why isolated: schema export/import is a pure read/write API that does not depend on bulk operations or versioning context. Phase 5's STI subclasses (Currency, Reference, File, Image, Percentage) are in place and the `type` column is the canonical discriminator — no Phase 5 dependency beyond the existence of those subclass files. Phase 1's `(entity_type, scope, parent_scope)` partition tuple is the idempotence key (RESEARCH §Schema export/import Anchor 3) — Phase 1 is shipped, so the dependency is satisfied.
</objective>

Locked decisions binding on this plan (from 06-CONTEXT.md):

- **Conflict policy:** `Field.import_schema(hash, on_conflict: :error)` keyword argument. Values: `:error` (default), `:skip`, `:overwrite`. Type changes always raise.
- **Idempotence key:** `(name, entity_type, scope, parent_scope)` for fields; `(code, entity_type, scope, parent_scope)` for sections. Using field name alone collapses two tenants' identically-named fields (CONTEXT §Key decision).
- **`schema_version: 1` envelope (Plan-time resolution per RESEARCH §Open-Question Resolutions):** export always includes `\"schema_version\" => 1` at the root. Import validates the value first; unrecognized versions raise `ArgumentError`.
- **Excludes data values:** export emits FIELD definitions and SECTION rows ONLY. NEVER emits typed Value rows. (CONTEXT §In scope.)
- **`default_value_meta` round-trip:** exported as the whole jsonb hash; imported via direct assignment to bypass re-cast (RESEARCH §Schema export/import Anchor 2).

Anchors confirmed in research:
- STI table is `typed_eav_fields`; `type` column is the STI discriminator (RESEARCH §Schema export/import Anchor 1). Export emits the literal `type` string.
- `Field::Base` validates name uniqueness scoped to `%i[entity_type scope parent_scope]` at field/base.rb:42–45 (RESEARCH §Schema export/import Anchor 3). This is the AR-native idempotence guarantee.
- `TypedEAV::Option` has `field_id`, `label`, `value`, `sort_order` (RESEARCH §Schema export/import Anchor 5). Import paths use `field.field_options.create!` after creating/updating the field row.
- Section uniqueness scoped to `%i[entity_type scope parent_scope]` on `code` (RESEARCH §Schema export/import Anchor 6).
- Section columns: `name`, `code`, `entity_type`, `scope`, `parent_scope`, `sort_order`, `active`.
- Field columns to export: `name`, `type`, `entity_type`, `scope`, `parent_scope`, `required`, `sort_order`, `field_dependent`, `options` (jsonb), `default_value_meta` (jsonb).

⚠ Discrepancy awareness:
- For `:overwrite` of a select/multi_select field, options must be DELETED then recreated (`Option` validates uniqueness of `value` per `field_id` — option.rb:12; an in-place update could fail uniqueness mid-update). RESEARCH §Risks confirms this approach.
- For `:skip`, options are LEFT UNTOUCHED (intentional — the policy is "ensure schema present, do not modify").
- `Field::File` shadow hazard: do NOT use bare `File` for any stdlib file I/O. The export/import code path does not touch the filesystem, but if a test fixture needs to write JSON to a temp file, use `::File` or use `Tempfile` (which doesn't shadow). RESEARCH §Risks §Field::File shadows ::File.
- The `validate_default_value` callback fires when a field is saved — when import overwrites a field whose `default_value_meta` includes a value that is no longer valid (e.g., out of range after `options` change), the save will fail. This is correct behavior — surface the AR error and let the caller resolve it. The plan's import does NOT silence validation failures.

  - Wraps the entire import in a single `ActiveRecord::Base.transaction do ... end` block — outer atomicity. If any field/section save fails (or `:error` policy fires), the entire import rolls back.
  - For each field entry in `hash["fields"]`:
    - Looks up existing via `Field::Base.find_by(name: entry["name"], entity_type: entry["entity_type"], scope: entry["scope"], parent_scope: entry["parent_scope"])`.
    - If existing AND existing.type != entry["type"]: raise `ArgumentError` with the type-swap message (must-haves §STI type-swap).
    - If existing AND `on_conflict == :error`: raise `ArgumentError, "Field '#{entry['name']}' already exists for #{entry['entity_type']} (scope=#{entry['scope']}, parent_scope=#{entry['parent_scope']}). Pass on_conflict: :skip or :overwrite to import over an existing schema."`
    - If existing AND `on_conflict == :skip`: increment `skipped`, continue (DO NOT touch field_options).
    - If existing AND `on_conflict == :overwrite`: update non-key attributes (`required`, `sort_order`, `field_dependent`, `options`, `default_value_meta`) via `field.assign_attributes(...)` then `field.save!`; for select/multi_select, `field.field_options.destroy_all` then create from `entry["options_data"]`. Increment `updated`.
    - If NOT existing: create via `Field::Base.create!(entry.except("options_data"))` — AR resolves the STI subclass from the `"type"` column string. For select/multi_select, after create, iterate `entry["options_data"]` and `field.field_options.create!(label:, value:, sort_order:)`. Increment `created`.
  - For each section entry in `hash["sections"]`: same logic with idempotence key `(code, entity_type, scope, parent_scope)`. Sections have no nested options.
  - Returns Hash: `{"created" => N, "updated" => N, "skipped" => N, "errors" => []}` (errors always empty on success — :error raises).
  - Inline comment above the method explaining the conflict-policy decision tree, the type-swap unconditional raise (data-loss guard), and the transaction wrapping rationale.
</acceptance>
